## 核心原则（来自 CLAUDE.md，不可覆盖）

CLAUDE.md 是项目的哲学基石。所有交互必须遵守：

- **创作永远没有绝对的正确。** 工具是辅助者，不是裁决者。
- **禁止**用理论否定用户创意、评判想法好坏、证明用户"想错了"
- **应该**理解用户意图，用知识滋养创意，帮用户走通自己的路
- 理论是工具不是答案，案例是参考不是模板

不要在 AGENTS.md 中重复 CLAUDE.md 内容——直接引用即可。

## 产品定位

本工具定位为**剧作老师**——不是项目管理者，也不是任务执行器。

**核心循环**：用户自由创作 → 后台观察/提取数据 → 判断进度和矛盾 → 温和提醒 → 用户决定 → 继续循环

- 不强制流程、不评判想法、不打断创作
- 五个模块（世界/角色/情节/主题/结构）螺旋迭代，不是线性完成

## 一、交互模型

### 主入口：ChatUI

用户主要通过 ChatUI 与工具聊天创作。支持：
- **渐进式提问**：由浅入深，每次只问一个问题，下一个问题是上一个答案的延伸
- **叠甲提醒**：专业建议裹在五层免责保护里，用户随时可忽略
- **心流模式**：几乎不打扰用户，最严重的问题也只亮卡片不插队
- **可配置响应频率**：用户可手动设置 Agent 主动说话的频率

### 画布UI（只读查看）

用户可切换到画布查看五大模块的结构化数据：世界观 / 角色 / 情节 / 主题 / 结构

### 卡片系统

用户可选添加监控卡片到侧边栏，实时显示 Agent 摘要——非侵入式，不打断创作。

### 排队消息

在当前对话间歇，按优先级排队提醒用户：
- P1: 用户主动触发的生成结果
- P2: 逻辑硬伤
- P3: 顽固模式（持续未解释的矛盾）
- P4: 重大涟漪（核心设定变更波及多个模块）
- P5: 普通提醒

### 独立报告

低频大块头输出，在保存时/下次打开时展示：更改汇总、节奏全局评估、灵感推荐整理。

## 二、Agent 架构

### 导播室 Hub（Agent Hub）

**所有 Agent 通过 Hub 中转，互不直接通信。** 如同片场的导演——每个部门只管干活，导演决定哪个画面优先、什么时候切。

Agent 只做两件事：
1. 向 Hub 提交"事件"（发现了什么）
2. 从 Hub 接收"指令"（现在该做什么）

这从根本上避免多 Agent 系统最常见的失败模式——Agent 间协调混乱（MAST 论文：37% 的多 Agent 失败来自此类问题）。

### Agent 全家福（12 个）

**第一层：响应式 Agent（用户说话即响应）**

| Agent | 角色比喻 | 职责 |
|-------|---------|------|
| 对话 Agent | 摄影指导 | 聊天、渐进引导、叠甲提醒、输出排队消息 |
| 提取 Agent | 灯光师 | 聊天→结构化数据（轻量实时 + 深度空闲） |
| 生成 Agent | 编剧组 | 大纲扩写、角色小传、场景描述 |
| 格式 Agent | DIT/输出 | Fountain 格式转换，有硬门禁否决权 |

**第二层：主动式状态机（后台常驻）**

| Agent | 角色比喻 | 职责 |
|-------|---------|------|
| 矛盾监控 Agent | 场记 | 四维度连续性检查 |
| 涟漪检测 Agent | 统筹 | 新内容影响扩散，触发相关模块更新 |
| 节奏分析 Agent | 剪辑指导 | 情绪曲线、结构评估，阶段报告 |
| 风格守护 Agent | 美术指导 | 跨章节语气一致性、角色对话风格追踪 |
| 伏笔追踪 Agent | 剧本统筹 | 播种→提醒→回收全生命周期管理 |
| 灵感推荐 Agent | 资料库 | 从知识库找滋养创意的参考（案例/理论/旧项目） |
| 时机管理 Agent | 副导演 | 层甲决策、心流检测、响应频率控制 |
| 记忆管理 Agent | 场记组长 | 用户偏好更新、空闲总结、更改汇总 |

**⚠️ 已知风险**：12 个 Agent 数量偏高。MAST 论文显示多 Agent 系统正确率仅 33-50%。建议先实现核心 5-6 个（对话+提取+矛盾监控+灵感推荐+时机管理+记忆管理），验证可行后逐步扩展。

## 三、核心机制

### 四维度连续性检查

| 维度 | 默认权重 | 先锋艺术响应 | 检测项 |
|------|---------|-------------|--------|
| 角色连续性 | 90% | 保留 | 属性、关系、知识状态、能力、位置变化 |
| 时间线连续性 | 85% | 保留 | 事件顺序、时间跨度、年龄推算 |
| 世界观一致性 | 75% | 降权到40% | 规则违反、设定冲突、科技水平 |
| 情节连续性 | 70% | 降权到35% | 因果链、伏笔、激励事件、角色目标 |

### 顽固模式检测

单次异常 → 轻甲随口一问；持续出现且无解释 → 升级为中/重甲提醒；严重逻辑矛盾 → 红灯+排队消息。

### 叠甲提醒（五层）

1. 免责声明（"按传统商业剧本的视角..."）
2. 具体事实，不下结论
3. 可能原因，不是唯一答案
4. 后果展示，非评判
5. 退路保留（"如果是先锋艺术，这些框架不适用"）

厚度由用户档位 + 问题主观程度 + 严重性共同决定。

### 格式否决权（硬门禁）

格式 Agent 在导出时拦截纯技术格式问题（缺少角色名、边距错误等）。这是纯技术检查，绝不涉及创作评判。无法通过 → 无法导出，但可一键自动修复。

### 心流模式

最高优先级的静默状态。再严重的问题也只亮卡片，不插队、不排队、不弹窗。

### 螺旋迭代与版本

- 用户自己决定保留什么（可能只留剧本名）
- 版本是生长，不是删改
- 可从任意历史版本"分叉"开新分支
- 跨分支比较和部分合并

## 四、知识库

### 技术底座：RAG-Anything

HKUDS 实验室开源（MIT），多模态知识图谱 RAG：
- 文档摄入 → 三层知识图谱构建（实体层 + 事件因果层 + 主题抽象层）
- 混合检索（向量相似度 + 图遍历）
- 这是从根本上区别于其他剧本工具的核心——不只是"找到相似段落"，而是理解叙事中的因果和关系

### 知识库内容（待详细设计）

- 故事库：用户旧项目存档
- 剧作理论库：麦基、坎贝尔、菲尔德等
- 规则库：专业知识体系
- 案例库：公共领域剧本、大纲等
- 视觉参考库：预留接口，前期不实现

## 五、用户记忆文档

区分于知识库——这是"关于用户"的数据：

- **用户级（跨项目）**：创作节奏、时段偏好、术语熟悉度、互动习惯
- **项目级（单项目）**：各分析维度权重、已免疫的提醒类型、薄弱环节标记

驱动 Agent 的自适应行为——什么时候提醒、用什么语言、叠几层甲。

## 六、数据库

四种并存（不是单一 SQLite）：

- **向量数据库**：RAG 知识库检索
- **JSON 数据库**：五大模块数据（世界/角色/情节/主题/结构）+ 知识库内容
- **传统数据库**：项目元数据、结构化配置
- **内存数据库**：热数据缓存

### 统一状态文件

所有 Agent 读写同一份 `story_state.json`（单一真相源），通过 Hub 中转确保一致性。

## OpenSpec 工作流

项目使用 OpenSpec（v1.3.1）管理变更，schema 为 `spec-driven`。四条核心命令：

| 命令 | 用途 |
|------|------|
| `/opsx:explore` | 探索想法、理清需求——只思考不实现 |
| `/opsx:propose` | 创建变更提案（proposal + design + tasks） |
| `/opsx:apply` | 按 tasks.md 逐项实现 |
| `/opsx:archive` | 完成后归档到 `openspec/changes/archive/` |
