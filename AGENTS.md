**当前处于设计阶段，没有任何实现代码。** 仓库内容为设计文档和 OpenSpec 骨架，不可修改 CLAUDE.md 中的核心原则。

> 2026-06-04 完成架构探索阶段，以下为设计共识汇总。

## 核心原则（来自 CLAUDE.md，不可覆盖）

CLAUDE.md 是项目的哲学基石。所有交互必须遵守：

- **创作永远没有绝对的正确。** 工具是辅助者，不是裁决者。
- **禁止**用理论否定用户创意、评判想法好坏、证明用户"想错了"
- **应该**理解用户意图，用知识滋养创意，帮用户走通自己的路
- 理论是工具不是答案，案例是参考不是模板

不要在 AGENTS.md 中重复 CLAUDE.md 内容——直接引用即可。

## 产品定位

本工具定位为**剧作老师**——不是项目管理者，也不是任务执行器。

**核心循环**：用户自由创作 → 后台观察/提取数据 → 判断进度和矛盾 → 温和提醒 → 用户决定 → 继续循环

- 不强制流程、不评判想法、不打断创作
- 五个模块（世界/角色/情节/主题/结构）螺旋迭代，不是线性完成

## 一、交互模型

### 主入口：ChatUI

用户主要通过 ChatUI 与工具聊天创作。支持：
- **渐进式提问**：由浅入深，每次只问一个问题，下一个问题是上一个答案的延伸
- **叠甲提醒**：专业建议裹在五层免责保护里，用户随时可忽略
- **心流模式**：几乎不打扰用户，最严重的问题也只亮卡片不插队
- **可配置响应频率**：用户可手动设置 Agent 主动说话的频率

### 画布UI（只读查看）

用户可切换到画布查看五大模块的结构化数据：世界观 / 角色 / 情节 / 主题 / 结构

### 卡片系统

用户可选添加监控卡片到侧边栏，实时显示 Agent 摘要——非侵入式，不打断创作。

### 排队消息

在当前对话间歇，按优先级排队提醒用户：
- P1: 用户主动触发的生成结果
- P2: 逻辑硬伤
- P3: 顽固模式（持续未解释的矛盾）
- P4: 重大涟漪（核心设定变更波及多个模块）
- P5: 普通提醒

### 独立报告

低频大块头输出，在保存时/下次打开时展示：更改汇总、节奏全局评估、灵感推荐整理。

## 二、Agent 架构

### 导播室 Hub（Agent Hub）

**所有 Agent 通过 Hub 中转，互不直接通信。** 如同片场的导播台/接线台——每个部门只管干活，Hub 只管理消息流转、权限和时机，不评价创作内容。

Agent 只做两件事：
1. 向 Hub 提交"事件"（发现了什么）
2. 从 Hub 接收"指令"（现在该做什么）

这从根本上避免多 Agent 系统最常见的失败模式——Agent 间协调混乱（MAST 论文：37% 的多 Agent 失败来自此类问题）。

### 架构收敛：薄 Hub + 单一用户出口 + 专职 Agent

> 2026-06-06 讨论收敛：Hub 保留，但必须做薄；对话 Agent 是唯一用户出口，但不能变成总控大脑；其他 Agent 坚持单一职责。

当前推荐结构命名为：**Thin Runtime Hub + Dialogue Gateway + Specialist Agents**。

V1 不再表述为"8 个并列自治 Agent"，而表述为：**核心 Agent + 按需能力 + 导出门禁**。Agent 名单不等于每轮触发名单；默认原则是按需触发、低频批处理、可降级熔断。

```
用户
 │
 ▼
对话 Agent（Dialogue Gateway）
只负责：聊天、渐进引导、叠甲表达、把最终内容说给用户
 │
 ▼
薄 Hub / Runtime
只负责：路由、调度、队列、权限、状态治理、成本控制
 │
 ├── 核心 Agent
 │   ├── 提取 Agent：对话/文本 → 结构化故事状态
 │   ├── 记忆管理 Agent：用户偏好、忽略规则、项目总结、状态投影
 │   └── 矛盾监控 Agent：连续性硬冲突检查，只提交证据
 │
 ├── 按需 / 低频能力
 │   ├── 生成能力：用户主动触发的一键生成、扩写、备选方向
 │   ├── 灵感检索：用户请求或空闲时提供案例/理论/旧项目参考
 │   └── 涟漪检查：核心设定变更后的影响范围分析，先不独立常驻
 │
 ├── 策略器
 │   └── Timing Policy：输出打扰风险、提醒厚度、展示模式
 │
 └── 导出门禁
     └── Format Gate：导出时强制检查 Fountain 等纯技术格式
```

#### Hub 的职责边界

Hub 是**运行时接线员**，不是创作裁判。它管理系统交通，不管理创作内容。

**Hub 可以做：**

- 路由：根据事件类型把任务交给对应 Agent。
- 调度：处理定时任务、空闲任务、保存后任务、下次打开提醒。
- 队列：根据 P1-P5、心流模式、冷却时间、用户频率偏好决定排队、延迟或静默。
- 权限：确保只有对话 Agent 能生成用户可见话术。
- 状态治理：维护事件日志、版本号、幂等键、失败重试、恢复重放。
- 成本控制：维护 token 预算、并发限制、超时和降级策略。
- 机械合并：合并重复事件、清理过期事件、维护事件生命周期。

**Hub 禁止做：**

- 直接对用户说话。
- 生成创作建议文案。
- 判断用户创意好坏。
- 把矛盾发现解释成唯一正确答案。
- 自行改写剧情、角色、主题或结构。
- 绕过对话 Agent 推送提醒。
- 在没有用户授权时做不可逆内容变更。

#### 对话 Agent 的职责边界

对话 Agent 是**唯一用户出口**，不是系统总控。

**对话 Agent 可以做：**

- 与用户聊天、渐进式提问、理解用户意图。
- 把结构化事件翻译成温和、可忽略、非裁判式的用户语言。
- 按叠甲规则调整表达厚度。
- 展示排队消息、卡片摘要、独立报告的用户可见文本。

**对话 Agent 禁止做：**

- 绕过 Hub 私下调度后台 Agent。
- 覆盖 Hub 的队列、冷却、心流模式和权限规则。
- 自己承担提取、矛盾检测、涟漪分析、伏笔追踪等专业职责。
- 以"正确答案"姿态裁决用户创作。

#### 时机管理拆分

时机管理不作为完整自治创作 Agent，而拆成两层：**Timing Policy** 负责语义判断，**Hub Runtime** 负责机械执行。这样避免 Hub 变成裁判，也避免对话 Agent 变成 god-object。

```
后台 Agent / 按需能力发现问题
        │
        ▼
Timing Policy
判断语义打扰风险：
"这是不是适合提醒？"
"用户是不是在心流？"
"该轻甲、中甲、重甲，还是只亮卡片？"
输出：delivery_mode / interrupt_risk / armor_level
        │
        ▼
Hub Runtime
执行机械规则：
排队 / 延迟 / 静默 / 允许输出
        │
        ▼
对话 Agent
生成用户可见表达
```

- **Timing Policy**：负责语义判断，如心流识别、提醒厚度、打扰风险。它是策略器，不是完整创作 Agent。
- **Hub Runtime**：负责机械执行，如队列优先级、冷却时间、定时触发、频率限制。
- **对话 Agent**：负责最终表达，不拥有最终调度权。

#### 结构化事件契约

后台 Agent 不应输出散乱自然语言给 Hub。所有发现必须先转成结构化事件，方便排队、恢复、评估和叠甲表达。

最小事件字段：

```json
{
  "type": "timeline_contradiction",
  "severity": "P2",
  "confidence": 0.82,
  "evidence": ["姐姐十年前死亡", "姐姐昨天打电话"],
  "affected_modules": ["character", "plot"],
  "suggested_delivery_mode": "queued_gentle_reminder"
}
```

字段含义：

- `type`：事件类型，如时间线矛盾、角色状态冲突、伏笔到期、灵感推荐。
- `severity`：P1-P5 优先级，不等于创作价值判断。
- `confidence`：Agent 对发现的置信度，用于降噪和评估。
- `evidence`：事实证据，只列观察到的内容，不下结论。
- `affected_modules`：影响的五大模块：世界 / 角色 / 情节 / 主题 / 结构。
- `suggested_delivery_mode`：建议展示方式，如排队提醒、只亮卡片、独立报告、静默记录。

#### 必须防止的架构漂移

1. **薄 Hub 变胖**
   - 风险：Hub 开始写提醒文案、解释矛盾、判断创意，变成隐形裁判。
   - 防线：Hub 只处理结构化字段；所有用户语义表达必须经过对话 Agent。

2. **对话 Agent 变成上帝对象**
   - 风险：对话 Agent 同时负责聊天、调度、状态判断、后台调用、提醒优先级，导致不可测试、不可替换、提示词膨胀。
   - 防线：对话 Agent 只能请求 Hub，不能私下调度后台 Agent；专业判断必须由专职 Agent 输出。

3. **后台 Agent 输出太散**
   - 风险：Agent 直接输出自然语言 findings，Hub 无法稳定排队，对话 Agent 无法稳定叠甲。
   - 防线：所有后台输出必须符合结构化事件契约。

4. **多 Agent 轻提醒叠加成打扰**
   - 风险：每个 Agent 都觉得自己只是轻轻提醒，合起来持续打断用户。
   - 防线：Hub 维护全局打扰预算；Timing Policy 统一判断打扰风险；心流模式优先级最高。

5. **误报累积损害信任**
   - 风险：矛盾监控、涟漪检测、伏笔追踪把先锋写法或故意留白误判为问题。
   - 防线：用户可标记"忽略此类提醒"；记忆管理 Agent 写入项目级偏好；相关提醒降权而非永久删除。

6. **状态恢复复杂度被低估**
   - 风险：队列、版本、分支、后台任务并存时，崩溃后无法恢复到一致状态。
   - 防线：Hub 必须维护事件日志、幂等键、任务状态和恢复重放策略；自动存档默认 10 分钟一次。

7. **Agent 名义单一职责，实际提示词重叠**
   - 风险：涟漪、矛盾、伏笔都检查因果关系，职责边界变模糊。
   - 防线：用输入/输出契约切边界，而不是只靠 Agent 名字区分。

### V1 Agent / Capability 收敛

V1 按**核心 Agent + 按需能力 + 策略器 + 导出门禁**组织，不把所有能力都做成常驻自治 Agent。

#### 核心 Agent（创作期高频 / 必须稳定）

| Agent | 角色比喻 | 职责 | 触发方式 |
|-------|---------|------|----------|
| 对话 Agent | 摄影指导 | 聊天、渐进引导、叠甲表达、输出排队消息 | 用户交互时 |
| 提取 Agent | 灯光师 | 对话/文本 → 结构化故事状态（轻量实时 + 深度空闲） | 每轮轻量，空闲深度 |
| 记忆管理 Agent | 场记组长 | 用户偏好、忽略规则、项目总结、状态投影 | 保存/空闲/会话结束 |
| 矛盾监控 Agent | 场记 | 四维度连续性硬冲突检查，只提交证据 | 核心状态变更后 / 保存后 |

#### 按需 / 低频能力（不常驻）

| 能力 | 角色比喻 | 职责 | 触发方式 |
|------|---------|------|----------|
| 生成能力 | 编剧组 | 一键生成故事、扩写、备选方向；只在用户授权后产出草稿 | 用户主动请求 / 点击 |
| 灵感检索 | 资料库 | 从知识库找滋养创意的参考（案例/理论/旧项目） | 用户请求 / 空闲低频 |
| 涟漪检查 | 统筹 | 核心设定变更后的影响范围分析，只列受影响范围，不自动修复 | 关键设定变更后 / 用户请求 |

#### 策略器与门禁

| 模块 | 角色比喻 | 职责 | 触发方式 |
|------|---------|------|----------|
| Timing Policy | 副导演 | 判断打扰风险、提醒厚度、展示模式；输出策略字段，不写用户话术 | 后台发现待展示事件时 |
| Format Gate | DIT/输出 | 导出时拦截 Fountain 等纯技术格式问题，有硬门禁否决权 | 导出时强制 |

#### 延后独立化的能力

| 能力 | 延后理由 |
|------|----------|
| 伏笔追踪 | 对长篇和复杂项目有价值，但依赖足够剧情长度；V1 先由记忆/矛盾/涟漪弱覆盖部分线索影响。 |
| 节奏分析 | 更偏专业阶段报告，需要足够文本体量后才有稳定价值。 |
| 完整风格守护 | 易误判用户实验性写法；V1 先做生成链路里的风格适配能力，不做后台巡逻式风格裁判。 |

**⚠️ 已知风险**：Agent 数量不是唯一风险，真正风险是常驻触发、状态恢复和跨 Agent 协调。V1 必须坚持：默认不运行、事件触发、低频批处理、可熔断；所有后台能力失败时不得阻断用户创作。

## 三、核心机制

### 四维度连续性检查

| 维度 | 默认权重 | 先锋艺术响应 | 检测项 |
|------|---------|-------------|--------|
| 角色连续性 | 90% | 保留 | 属性、关系、知识状态、能力、位置变化 |
| 时间线连续性 | 85% | 保留 | 事件顺序、时间跨度、年龄推算 |
| 世界观一致性 | 75% | 降权到40% | 规则违反、设定冲突、科技水平 |
| 情节连续性 | 70% | 降权到35% | 因果链、伏笔、激励事件、角色目标 |

### 顽固模式检测

单次异常 → 轻甲随口一问；持续出现且无解释 → 升级为中/重甲提醒；严重逻辑矛盾 → 红灯+排队消息。

### 叠甲提醒（五层）

1. 免责声明（"按传统商业剧本的视角..."）
2. 具体事实，不下结论
3. 可能原因，不是唯一答案
4. 后果展示，非评判
5. 退路保留（"如果是先锋艺术，这些框架不适用"）

厚度由用户档位 + 问题主观程度 + 严重性共同决定。

### 格式否决权（硬门禁）

Format Gate 在导出时拦截纯技术格式问题（缺少角色名、边距错误等）。它属于导出流程门禁，不属于创作期常驻 Agent。这是纯技术检查，绝不涉及创作评判。无法通过 → 无法导出，但可一键自动修复。

### 心流模式

最高优先级的静默状态。再严重的问题也只亮卡片，不插队、不排队、不弹窗。

### 螺旋迭代与版本

- 用户自己决定保留什么（可能只留剧本名）
- 版本是生长，不是删改
- 可从任意历史版本"分叉"开新分支
- 跨分支比较和部分合并

## 四、知识库

### 技术底座：RAG-Anything

HKUDS 实验室开源（MIT），多模态知识图谱 RAG：
- 文档摄入 → 三层知识图谱构建（实体层 + 事件因果层 + 主题抽象层）
- 混合检索（向量相似度 + 图遍历）
- 这是从根本上区别于其他剧本工具的核心——不只是"找到相似段落"，而是理解叙事中的因果和关系

### 知识库内容（待详细设计）

- 故事库：用户旧项目存档
- 剧作理论库：麦基、坎贝尔、菲尔德等
- 规则库：专业知识体系
- 案例库：公共领域剧本、大纲等
- 视觉参考库：预留接口，前期不实现

## 五、用户记忆文档

区分于知识库——这是"关于用户"的数据：

- **用户级（跨项目）**：创作节奏、时段偏好、术语熟悉度、互动习惯
- **项目级（单项目）**：各分析维度权重、已免疫的提醒类型、薄弱环节标记

驱动 Agent 的自适应行为——什么时候提醒、用什么语言、叠几层甲。

## OpenSpec 工作流

项目使用 OpenSpec（v1.3.1）管理变更，schema 为 `spec-driven`。四条核心命令：

| 命令 | 用途 |
|------|------|
| `/opsx:explore` | 探索想法、理清需求——只思考不实现 |
