# 架构总览

> 这份文档回答一个问题:**这个工具整体长什么样?**
> 它讲"由哪几大块组成、各块负责什么、怎么拼在一起",
> 不讲"用什么技术框架、代码怎么写"(那是技术选型,另行记录)。
>
> - 哲学铁律见 `CLAUDE.md`
> - 架构共识总纲见 `AGENTS.md`
> - 各机制的详细设计见 `机制设计.md`

---

## 一句话定位

这是一个**剧作老师**,不是项目管理器,也不是任务执行器。

它的核心循环:

```
用户自由创作 → 后台默默观察/提取 → 判断进度和矛盾
   → 温和提醒(可忽略) → 用户决定 → 继续循环
```

不强制流程、不评判想法、不打断创作。

---

## 最高指导原则:留余地,不下定论

> 这一条凌驾于所有模块之上。任何模块的设计若与它冲突,以它为准。

```
┌─────────────────────────────────────────────┐
│  系统永远"留余地",从不"下定论"。              │
│  判断权,永远在用户手里。                       │
│                                               │
│  系统主动开口时 → 闭嘴、压薄、只递证据         │
│  用户主动开口时 → 全力配合、有多少给多少       │
└─────────────────────────────────────────────┘
```

它在每个模块里的体现:

| 模块 | 怎么体现"留余地" |
|------|------------------|
| 结构化事件 | 后台只提交"证据",无权决定打扰(权限写进数据结构) |
| 格式理解 | 只报"有什么+缺什么",不建议怎么补 |
| 用户记忆 | 老规则"沉底不删",降权而非删除 |
| 叠甲厚度 | 系统主动说话压薄(防说教),用户主动问才放开 |
| 版本分叉 | "生长不覆盖",节点永不丢弃 |
| 涟漪检查 | 只列事实关联,不下"必须改"的结论 |
| 心流模式 | 系统主动全闭嘴,用户主动求助立刻响应 |
| 渐进提问 | 提问是"邀请"不是"考试",用户随时可不答 |
| 顽固模式 | 你坚持的就是你要的——反复出现转为尊重 |

一句话:**别的 AI 工具"我比你懂我帮你定",这个工具"你比我懂你的故事,我只递证据"。**

## 后端:对话网关 + 薄 Hub + 专职 Agent

这是整个系统的核心结构。设计原则:**薄 Hub 收敛 + 单一用户出口 + 专职 Agent**。

```
用户
 │
 ▼
对话 Agent(Dialogue Gateway)= 唯一用户出口
只负责:聊天、渐进引导、叠甲表达、把内容说给用户
 │
 ▼
薄 Hub / Runtime = 运行时接线员
只负责:路由、调度、队列、权限、状态治理、成本控制
(不直接对用户说话,不评价创作好坏)
 │
 ├── 核心 Agent(高频,必须稳定)
 │   ├── 提取 Agent ── 对话/文本 → 结构化故事状态(系统的"眼睛")
 │   ├── 记忆管理 Agent ── 用户偏好、忽略规则、项目总结
 │   └── 矛盾监控 Agent ── 连续性硬冲突检查,只提交证据
 │
 ├── 按需 / 低频能力(默认不运行,事件触发)
 │   ├── 生成能力 ── 用户主动触发的生成、扩写、备选方向
 │   ├── 灵感检索 ── 案例/理论/旧项目参考
 │   ├── 涟漪检查 ── 核心设定变更后的影响范围分析
 │   └── 格式理解工具 ── 读剧本格式标记(被矛盾监控/提取/Format Gate 调用)
 │
 ├── 策略器
 │   └── Timing Policy(副导演)── 判断打扰风险、提醒厚度、展示模式
 │
 └── 导出门禁
     └── Format Gate(DIT/输出)── 导出时强制检查 Fountain 等技术格式
```

### 为什么是这个结构?

后台 Agent **互不直接通信**,全部通过 Hub 中转。这从根本上避免多 Agent 系统最常见的失败模式——Agent 间协调混乱。

三个角色的职责边界(谁能做什么):

- **对话 Agent**:唯一能生成"用户可见话术"的角色。但不能私自调度后台 Agent,不能承担专业判断。
- **薄 Hub**:只处理结构化字段和系统交通。禁止直接对用户说话、禁止评判创作。
- **专职 Agent**:只做单一职责的活,只向 Hub 提交结构化事件,从 Hub 接收指令。

### 时机管理拆成两层

```
后台 Agent 发现问题
        │
        ▼
Timing Policy(语义判断)
"该不该提醒?用户在心流吗?该几层甲?"
输出:delivery_mode / interrupt_risk / armor_level
        │
        ▼
Hub Runtime(机械执行)
排队 / 延迟 / 静默 / 允许输出
        │
        ▼
对话 Agent(最终表达)
生成用户可见话术
```

这样避免 Hub 变成裁判,也避免对话 Agent 变成什么都管的"上帝对象"。

## 数据库:四种并存 + 事件日志真相源

| 数据库类型 | 装什么 |
|------------|--------|
| 事件日志(真相源) | append-only event log,所有改动只追加不擦旧 |
| 向量数据库 | RAG 知识库检索(LightRAG) |
| JSON 数据库 | 五大模块数据的**当前投影**(`story_state.json`)+ 知识库内容 |
| 传统数据库 | 项目元数据、结构化配置 |
| 内存数据库 | 热数据缓存 |

**真相源是 append-only event log**:事实源是事件日志,每次改动只追加新记录、永不擦旧;`story_state.json` 是由事件日志重建出来的**当前状态投影**,用于快速读取和 Agent 工作上下文。所有 Agent 通过 Hub 中转读写,确保一致性。

> 为什么是事件日志而非单一可变文件:本产品天然需要版本、生长、分叉、忽略、恢复、重放、事件排队——append-only 模型完美契合这些需求。投影崩溃可随时从日志重建,恢复靠重放已记录的事件而非重跑 LLM。

## 知识库:LightRAG(图谱增强 RAG)

不只是"找相似段落",而是理解叙事中的因果和关系(实体-关系抽取 + 双层检索 + 图遍历)。

⚠️ **重要限制**:知识图谱对"理解叙事因果"在内向型叙事(心理描写、情感驱动、文艺片)上会退化。因此 V1 把图谱检索定位为**灵感检索/低频报告**,不进入实时矛盾判断主链路。

## 五大模块(创作内容的组织维度)

世界观 / 角色 / 情节 / 主题 / 结构。

它们**螺旋迭代,不是线性完成**——用户随时在五个模块间来回,系统跟着观察哪块最空、哪块有矛盾。

---

## V1 边界:不做什么

为防止架构漂移和过度工程,V1 明确**不做**:

- 不做常驻自治的伏笔追踪 Agent(由记忆/矛盾/涟漪弱覆盖)
- 不做后台巡逻式的节奏分析 / 风格裁判
- 不做版本的自动语义合并(风险极高,只做手动 + 矛盾监控护航)
- 所有后台能力**失败时不得阻断用户创作**(可熔断、可降级)
