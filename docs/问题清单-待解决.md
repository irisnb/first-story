## ⚠️ 今天审计暴露的坏习惯（写下来，时时自检）

2. **把"写了更多文档"当成"离产品更近"**——今天产出 4 个文档还提交了，但 `openspec/changes/` 和 `openspec/specs/` 两个目录都是空的，OpenSpec 工作流根本没真正启动。文档厚度 ≠ 规格硬度。

4. **横向铺开 10 个机制，不肯纵向打穿一条闭环**——产品最小价值只需一条线能跑（见下方"最小闭环"），机制再精细，这条线不通就是空中楼阁。
5. **"共识"定义太软**——把"我理解它该是什么感觉"误当成"系统知道它该怎么运行"。

## 🎯 转向决策：停止加功能，定死现有的，开始实现

**已定共识（够了，不再扩）**：哲学、产品定位、薄 Hub 架构方向、10 个机制的"态度"和"愿景"。这些冻结，作为产品愿景保留，**不再逐个精修**。

**接下来的关键路径（按顺序，别跳）**：

1. **🔴 解决真相源矛盾**（见 P0-0）——今天就该定，全局统一。
2. **定义最小数据模型**——只做 `角色 / 情节事件 / 事实 / 矛盾事件 / 用户偏好` 五个对象，先给字段；主题和结构先放空壳（它们太抽象，会把人又拽回哲学设计）。
3. **写第一个真正的 OpenSpec change**：`define-minimal-story-state-and-event-log`，只装最小闭环，不碰 10 个机制全量。
4. **把 10 个机制狠切成 MVP / Later**（见下方切分表）。
5. **冻结一个"够用"的技术栈**（不是最优）——本地 Web + 简单后端 + event log + SQLite；LightRAG / LangGraph / 多数据库全部后置。

**MVP 最小闭环（产品成立的唯一证明）**：
```
用户写一段创作
  → 系统提取结构化事实（角色/事件）
  → 写入 event log + 重建 story_state 投影
  → 检测一个硬矛盾（如"姐姐十年前死亡却昨天打电话"）
  → 生成一条温和提醒
  → 用户选择忽略/接受/保留
  → 系统记住这个偏好（降权）
```
这条线能跑，产品就成立。跑不通，10 个机制都是空中楼阁。

**10 个机制 MVP / Later 切分**：

| 机制/能力 | V1 MVP | 说明 |

| 事件契约 #1 | ✅ 必须 | 所有 Agent 输出的基础 |
| 提取 Agent #7 | ✅ 必须 | 没提取就没状态 |

### ✅ 0. 真相源矛盾 —— 已对齐（2026-06-09 落实）

**问题**：同一个"谁是单一真相源"的根决策，三份文档说法不一致：
- `AGENTS.md` + `架构总览.md`：说 `story_state.json` 是单一真相源（所有 Agent 读写同一份）。
- 本清单 P0-2：说已改成 **append-only event log 才是真相源**，`story_state.json` 降为"可重建投影"。

**为什么是地基的地基**：真相源没定，下面这些全无法稳定设计——崩溃恢复 / 版本分叉 / 状态重放 / Agent 幂等 / 用户撤销 / 自动存档 / 事件队列 / 多 Agent 一致性。

**建议拍板（Oracle 裁决，倾向此方案）**：
> 系统的事实源是 **append-only event log**；`story_state.json` 是由事件日志重建出来的**当前状态投影**，用于快速读取和 Agent 工作上下文。

理由：本产品天然需要版本、生长、分叉、忽略、恢复、重放、事件排队——event log 模型完美契合。

**待办**：把 `AGENTS.md`、`架构总览.md` 里"story_state.json 单一真相源"的表述，全部改成上面这一句口径，三处统一。

### ✅ 0b. 最小数据模型缺失 —— 已形成 MVP 施工图（2026-06-15 落实）

**问题**：五大模块（世界/角色/情节/主题/结构）被反复提及，`story_state.json` 被当真相源/投影，但**从来没有字段级定义**：角色有哪些属性？关系怎么建模？时间线怎么表示？事实怎么编码？一个矛盾事件引用哪些状态？

**为什么致命**：没有数据模型，叠甲、心流、顽固、涟漪这些上层机制全都"漂"在空中——底层字段一变，上层全跟着塌。这是审计抓出的"先盖保护层不砌承重墙"的根因。

**✅ 已完成（2026-06-15）**：`define-minimal-story-state-and-event-log` 已定义 5 个 MVP 对象的最小字段，并用 walkthrough + review cleanup 反查修正——
- `Character`（角色：id / 名字 / 状态 / 状态自哪个事件起 / 已知事实 / 关系）
- `PlotEvent`（情节事件：id / 摘要 / 时间锚点 / 参与者 / 断言的事实）
- `Fact`（事实：id / 内容 / 来源事件）
- `ContinuityEvent`（矛盾事件：id / type / severity / contradiction_confidence / evidence_fact_ids / status / delivery）
- `project_preferences`（项目偏好：降权规则 / 显式确认设定）
- 主题、结构先放**空壳**（太抽象，会把人拽回哲学设计）。

**剩余进入后续 change**：Story Clock 推断、`story_time.relative.anchor` 稳定锚定、置信度动态变化算法、跨项目用户记忆。

### 1. Token 成本管控 ⏳

**问题**：当前架构下，若用户每轮对话都唤醒多个后台 Agent、且每个都重读整个故事状态，token 成本将不可控。行业数据显示 MetaGPT 浪费 72%、CAMEL 浪费 86% 的 token；Anthropic 多 Agent 系统用了单次聊天的约 15 倍 token。

**✅ 已达成方向共识**：
- 成本工程化从第一天纳入（sopher.ai 验证：每 Agent 预算分配 + 实时成本追踪做成核心功能，非事后补）
- 三层模型分级（postwriter 验证：机械检查用便宜模型，创意判断才用最贵的）
- **死规矩先行**（Novel-OS / InkOS 验证：先跑不花钱的确定性检查，结果喂给 LLM 当上下文——省钱 + 降误报）
- 不是每轮全员出动：对话 + 轻量提取每轮跑，矛盾监控攒批（停笔/保存后）

**仍待决策**：
- 每个 Agent 的 token 上限具体数字？
- 状态分层加载策略（hot / recent / entity slice / cold archive）细节？
- 成本熔断后用户看到什么（降级体验）？

**参考来源**：MAST 论文、Anthropic 工程博客、Galileo 多 Agent 成本分析、sopher.ai / postwriter 实证

### 2. 状态持久化 + 错误恢复 ✅（方向已定）

**问题**：原设计把 `story_state.json` 当唯一真相源、所有 Agent 共写、每 10 分钟存档，存在并发覆盖、半截写入损坏、崩溃丢失用户创作的风险。LLM Agent 长任务可靠性递减极快（95% 单步 × 10 步 ≈ 60% 总成功率）。

**✅ 已达成方向共识（流水账本）**：
- 改用 append-only event log（流水账本）作真相源，每次改动只追加新记录，永不擦旧
- `story_state.json` 降为"可随时重建的投影"，不再是写入争用点
- 用户原始创作内容**每轮立即落盘**，不等 10 分钟
- 已接受的结构化状态恢复靠重放 event log，而非重跑 LLM；尚未提取的用户正文下次打开时补提取

**仍待设计**：
- 账本存储介质？投影快照频率？
- 存储介质、投影快照频率、日志轮转策略？
- 分支/合并：V1 只做"创建分支 + 对比 + 手动复制"，不承诺自动语义合并

> 🔗 **2026-06-09 已加固**：版本分叉机制已钻透（见 `机制设计.md` §5）——确认"生长不覆盖、节点永不丢弃"，比较 V1 只做文字对比，部分合并 V1 明确不做（蝴蝶效应风险），分支树整理用"命名主线不限制分叉"方案。本条 P0 的"分支/合并"决策已与之对齐。

### 3. 评估/验证框架 ✅（方向已定）

**问题**：74% 的生产 AI 团队依赖人工评估。我们需要机制回答"系统是否真的温柔不打扰、没乱报警、没把主观意见伪装成事实"——但不评判创作质量。

**✅ 已达成方向共识（专测系统行为）**：
- 只测**系统行为**，不测创作质量。四类指标：
  1. 结构提取准确率（提取到的角色/时间/关系对不对）
  2. 连续性提醒误报率（该提醒/不该提醒/只记录/需澄清）
  3. 哲学合规（话术里有没有"你应该""必须按三幕式"等裁判腔；有没有给"可能是故意的"退路）
  4. 运行时行为（崩溃恢复、事件去重、预算熔断、心流不插队、用户忽略后是否真降权）
- LightRAG 自带 RAGAS 评估，可复用一部分

**仍待设计**：具体指标阈值、人工标注数据集、评估频率

> 🔗 **2026-06-09 已加固**：今天钻透的机制给"测系统行为"提供了具体测试对象——叠甲厚度（§4）可测"系统主动时是否真压薄"；提取 Agent（§7）的待复核队列可测"低把握度是否真不直接写主状态"；心流识别（§8）可测"心流时是否真不插队"；顽固模式（§10）可测"反复出现是否真转向尊重、是否有给退路"。指标阈值待设计，但要测什么已清晰。

**参考来源**：MAP 研究（306 名生产 AI 从业者调查）、MAST 论文验证章节、LightRAG RAGAS

## P1 — 应该解决

### 4. 上下文窗口管理 ⏳

**问题**：Claude 上下文窗口在 65% 填满时开始退化，80% 以上开始主动矛盾之前的设定。长会话场景下上下文会逐渐腐化。Sudowrite 用户的核心投诉就是连续性丢失。

**需要决策**：
- 何时触发上下文重置？（按消息数？按 token 占比？按质量退化信号？）
- 什么信息必须前置加载到每个新会话？（故事圣经？当前角色状态？）
- 记忆管理 Agent 的压缩 vs 丢弃的决策逻辑？
- 是否需要类似 Claude Code 的 `/compact` 机制？

> 🔗 **2026-06-09 部分加固**：用户记忆机制已钻透（见 `机制设计.md` §3）——确立"老规则沉底不删、降权而非丢弃"原则。这给"压缩 vs 丢弃"定了基调：**倾向降权压缩，不做硬删除**。但"会话上下文窗口"(技术层 token 腐化)和"用户记忆文档"(产品层关于用户的数据)是两件事，本条 P1 关注的前者仍待独立决策。

**参考来源**：Sudowrite 用户反馈分析、Claude 长对话质量退化社区报告、Anthropic 上下文管理实践

### 5. 知识图谱对内向型叙事的局限性 ✅（方向已定）

**问题**：学术研究验证，知识图谱在动作导向叙事中有效，但在心理描写/情感驱动的内向型叙事中效果下降甚至逆转。剧本不全是动作片（爱情、家庭、文艺片、心理惊悚都高度依赖内部状态）。

**✅ 已达成方向共识**：
- 图谱检索定位为**灵感检索 / 低频报告**，不进入实时矛盾判断主链路
- 默认可先用普通语义检索，图谱仅作增强
- 技术底座已统一为 **LightRAG**（原 RAG-Anything 能力已合并，且 RAG-Anything 自 2026-05 起停止核心维护）

**仍待设计**：
- 叙事类型如何识别？（系统建议，不自动盖章）
- 是否给用户手动切换检索模式（动作/关系/内向/实验）？
- 内向叙事下重点索引什么（情绪状态、欲望、恐惧、关系变化、未说出口的信息）？

**参考来源**："Guiding Generative Storytelling with Knowledge Graphs" (2025)、E²RAG 论文、LightRAG v1.5.0rc3 发布说明

## P2 — 可后期优化

### 6. Agent 数量收敛 ✅（已收敛）

**结论**：已在 AGENTS.md 收敛为"核心 4 + 按需能力 + 门禁"，不再是 12 个并列自治 Agent。详见 AGENTS.md「架构收敛」与「V1 Agent / Capability 收敛」段。

**事实修正**：原文"MAST 显示 3-4 个 Agent 系统正确率仅 33-50%"为误植——33.33% 是 ChatDev 单个框架在 ProgramDev 基准上的最差表现，不是"3-4 个 Agent 系统"的普遍正确率。MAST（arXiv 2503.13657，UC Berkeley，NeurIPS 2025）的真正结论是：失败 41.77% 源于系统设计本身而非模型能力，好的 MAS 设计能在同一模型下显著提升表现——这反而是"薄 Hub 收敛"的依据，不是"Agent 越少越好"。

**外部佐证**：实际调研的 6 个创作类多 Agent 项目均落在 5-10 个 Agent 区间（Novel-OS 5、InkOS 5、sopher.ai 5、NovelPilot 9、postwriter 10），与我们的收敛区间一致。

**参考来源**：MAST 论文、创作类项目实证调研（2026-06-07）

### 7. 独立验证 Agent ⏳

**问题**：MAST 研究明确指出："The same agent that produced the output should not be the one verifying it." 我们没有专门验证 Agent。矛盾监控做了一部分但不等于完整验证。

**当前定位**：评估框架（P0-3）的"运行时行为测试"已覆盖部分验证需求。V1 可先不独立成 Agent，由评估框架 + 死规矩确定性检查弱覆盖；待系统稳定后再评估是否需要独立验证 Agent。

**仍待设计**：
- 验证 Agent 的检查维度？（输出一致性？任务完成度？）
- 验证失败后的处理？（回退？重试？通知用户？）

**参考来源**：MAST 论文 FC3 类别分析、Anthropic 验证模式
