# 分层摘要自动更新 - 设计文档

> 本文档记录对话上下文摘要的设计方案，供实现时参考。

---

## 一、功能概述

**目的**：当对话轮数增加时，自动从聊天记录中提炼摘要，帮助 LLM 在新上下文中快速进入状态。

**分层机制**：
- 小摘要（10 轮）：只更新 `recent_focus`
- 大摘要（30 轮）：全面更新 `world_brief`、`plot_brief`、`character_brief`、`recent_focus`

---

## 二、数据模型

### context_summary 字段

```python
# backend/app/models/state.py

@dataclass
class ContextSummary:
    """对话上下文摘要，用于帮助 LLM 快速进入状态。"""
    
    # 大摘要（30轮更新）
    world_brief: str = ""      # 世界观简述，200字以内
    plot_brief: str = ""       # 情节简述，200字以内
    character_brief: str = ""  # 角色简述，200字以内
    
    # 小摘要（10轮更新）
    recent_focus: str = ""     # 最近创作焦点，100字以内
    
    # 元信息
    last_minor_update: str | None = None   # 小摘要更新时间
    last_major_update: str | None = None   # 大摘要更新时间
    turn_count: int = 0                     # 当前对话轮数
```

### story_state.json 结构

```json
{
  "story": {
    "characters": [...],
    "facts": [...],
    "plot_events": [...],
    "context_summary": {

      "plot_brief": "主角小红...",
      "character_brief": "小红：主角...",
      "recent_focus": "最近在讨论...",
      "last_minor_update": "2026-06-21T01:00:00",
      "last_major_update": "2026-06-21T00:30:00",
```

### 对话 API 改动

```python

class DialogueAgent:
    def respond(self, message: str, ...) -> DialogueResult:
        # 1. 保存用户消息
        # 2. 调用 LLM 生成回复
        # 3. 保存助手消息
        # 4. 更新对话轮数
        self._increment_turn_count()
        
        # 5. 检查是否需要更新摘要
        turn_count = self._get_turn_count()

            self._schedule_major_summary_update()
        elif turn_count % 10 == 0:
```

## 四、摘要生成 API

### 端点设计

```
POST /api/v1/projects/{project_id}/context-summary/generate

Request:
{
  "type": "minor" | "major"  # 小摘要或大摘要
}

Response:
{
  "world_brief": "...",
  "plot_brief": "...",
  "character_brief": "...",
  "recent_focus": "..."
}
```

### LLM Prompt - 小摘要

```
你是创作助手。请根据最近 10 轮对话，提炼用户最近在创作什么。

## 最近 10 轮对话
{最近聊天记录}

请输出 JSON：
{
  "recent_focus": "最近创作焦点（100字以内，用户最近在讨论/创作什么）"
}

注意：
- 只关注最近的话题，不要回顾更早的内容
- 简洁，抓住核心
```

### LLM Prompt - 大摘要

```
你是创作助手。请根据以下信息生成项目摘要。

## 最近 30 轮对话
{最近聊天记录}

## 当前故事状态
- 角色：{角色列表}
- 事实：{事实列表}
- 情节事件：{情节事件列表}

请输出 JSON：
{
  "world_brief": "世界观简述（200字以内）",
  "plot_brief": "情节简述（200字以内）",
  "character_brief": "角色简述（200字以内）",
  "recent_focus": "最近创作焦点（100字以内）"
}

注意：
- 简洁、抓重点
- 突出用户明确表达的内容
- 如果某方面信息不足，写"暂无"
```

## 六、实现步骤

1. **数据模型**
   - 在 `backend/app/models/state.py` 添加 `ContextSummary` 类
   - 在 `StoryState` 中添加 `context_summary` 字段

2. **事件类型**
   - 新增 `context_summary.updated` 事件类型
   - 记录摘要更新历史

3. **摘要生成服务**
   - 创建 `backend/app/services/context_summary.py`
   - 实现小摘要和大摘要生成逻辑

4. **对话 API 改动**
   - 在 `DialogueAgent.respond()` 中添加轮数计数
   - 触发后台摘要更新任务
   - 在系统提示中注入摘要

5. **投影器改动**
   - `projector.py` 处理 `context_summary.updated` 事件
   - 更新 `story_state.json` 中的摘要字段

## 七、成本估算

```
小摘要（每 10 轮）：
  • 输入：~1000 tokens（10 轮对话）
  • 输出：~100 tokens
  • 小计：~1100 tokens

大摘要（每 30 轮）：
  • 输入：~5000 tokens（30 轮对话 + 故事状态）
  • 输出：~600 tokens
  • 小计：~5600 tokens

每 30 轮总成本：
  • 小摘要：2 次 × 1100 = 2200 tokens
  • 大摘要：1 次 × 5600 = 5600 tokens
  • 总计：7800 tokens
  • 平均每轮：260 tokens
```

---

## 八、注意事项

1. **摘要更新是后台任务**：不阻塞对话响应
2. **失败不影响对话**：摘要生成失败时，继续使用旧摘要
3. **首次打开项目**：如果没有摘要，可以立即生成一个初始摘要
4. **用户可手动触发**：提供"重新生成摘要"按钮（可选）
