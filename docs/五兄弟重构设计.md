# 五兄弟重构设计文档

> 本文档记录五模块（世界观/角色/情节/主题/结构）重构的技术方案，供下次实现时参考。

---

## 一、背景与问题

### 当前实现

```
chat.message → extraction → facts[] (flat list)
                          → plot_events[] (flat list)
                          → characters[] (flat list)
```

**问题**：
- 无分类，所有"事实"混在一起
- 无文档结构，只是条目列表
- 用户无法直接编辑
- 不符合用户预期的"不断完善的世界观文档"形式

### 用户预期

- 每个模块 = 一个 MD 文档，用户可直接编辑
- 系统自动整理/摘要，不什么都往里放
- 文档有总述 + 分项详述
- 分类判断需要专门的 API，不挤占主 chat API

---

## 二、核心决策

### 决策 1：真相源 = MD 文档

**选择**：MD 文档是用户可见的真相源

**数据流**：
```
MD 文档 = 用户编辑界面（主真相源）
    │
    ▼ 解析器（用户保存时触发）
    │
事件日志 = 结构化事件（派生，供 Agent 使用）
    │
    ▼ 投影
    │
story_state.json = 快速读取缓存
```

**理由**：
- 用户看到/编辑的是 MD（真相源）
- Agent 工作用的是事件日志（结构化）
- 两者通过解析器同步

### 决策 2：分类 API 分开

**选择**：独立的分类 API，每条消息触发

**理由**：
- 单一职责，prompt 更清晰
- 失败隔离
- 可用不同模型（分类用便宜模型）

**成本**：每条消息增加 ~550 tokens（约 70% 增加）

### 决策 3：固定 section + LLM 判断

**选择**：每个模块有预定义的 section 列表，LLM 判断内容放哪个 section

### 决策 4：乐观锁

**选择**：用户编辑时锁定文档，系统添加排队等待

### 决策 5：追加格式 = 列表项

**选择**：系统追加内容时，以列表项形式追加到对应 section

### 决策 6：MD 解析器 = AST 解析

**选择**：使用 remark 等 Markdown AST 解析库

**理由**：结构化程度高，一劳永逸

## 三、五模块 Section 定义

### 世界观 (world.md)

```markdown
# 世界观

## 总述
[一段话概括世界核心设定]

## 魔法/技术系统
- 规则、限制、代价

## 社会结构
```

```markdown
- 角色名：描述
```

## 五、API 设计

### 分类 API

```
POST /api/v1/projects/{project_id}/classify

Request:
{
  "content": "主角叫小红，是一名社畜，女 26岁",
  "context": {
    "world_summary": "动画+拼贴风格，上班族怨念具象化...",
    "character_summary": "小红：主角，社畜...",
    "plot_summary": "..."
  }
}

Response:
{
  "classifications": [
    {
      "module": "character",
      "section": "主要角色",
      "content": "小红：主角，女性，26岁，社畜",
      "confidence": 0.92
    },
    {
      "module": "world",
      "section": "社会结构",
      "content": "存在上班族群体",
      "confidence": 0.65
    }
  ]
}
```

### 文档 API

```
GET /api/v1/projects/{project_id}/modules/{module}
  - 返回模块 MD 文档内容

PUT /api/v1/projects/{project_id}/modules/{module}
  - 更新模块 MD 文档（用户编辑）

POST /api/v1/projects/{project_id}/modules/{module}/lock
  - 获取编辑锁

DELETE /api/v1/projects/{project_id}/modules/{module}/lock
  - 释放编辑锁
```

## 六、乐观锁设计

```
用户打开 MD 编辑器
     │
     ▼
前端：POST /modules/{module}/lock
     │
     ▼
后端：检查锁状态
     • 无锁 → 加锁（user_id, timestamp, TTL=5min）
     • 有锁且过期 → 强制拿锁
     • 有锁且未过期 → 返回 409 Conflict
     │
     ▼
用户编辑中...
     │
     ├── 每 30s 心跳续期锁
     │
     ▼
用户保存 / 关闭编辑器
     │
     ▼
前端：DELETE /modules/{module}/lock
     │
     ▼
后端：释放锁 + 处理排队中的系统添加


系统想添加内容
     │
     ▼
后端：检查锁状态
     • 无锁 → 直接追加
     • 有锁 → 加入队列，等锁释放
```

## 七、实现顺序

```
Phase 1: 基础设施
  ├── 定义五模块 section 结构
  ├── 实现 MD 解析器（remark）
  └── 实现 MD 渲染器（追加列表项）

Phase 2: 分类 API
  ├── 设计 prompt
  ├── 实现端点
  └── 测试分类准确率

Phase 3: 集成
  ├── 修改对话流程，每条消息触发分类
  ├── 分类结果写入 MD 文档
  └── MD 变更触发解析 → 事件日志

Phase 4: 乐观锁
  ├── 锁 API
  ├── 前端编辑器集成
  └── 队列处理

Phase 5: 前端 UI
  ├── 五模块文档编辑器
  ├── ModulesView 改造
  └── 文档预览
```

## 八、成本估算

```
每条 chat message 的 LLM 调用：

1. 对话 API（主接口）
   - 输入: ~500 tokens
   - 输出: ~300 tokens
   - 小计: ~800 tokens

2. 分类 API（独立接口）
   - 输入: ~400 tokens
   - 输出: ~150 tokens
   - 小计: ~550 tokens

总计: ~1350 tokens / 条消息

对比当前: ~800 tokens / 条消息
增加: ~550 tokens / 条消息 (~70% 增加)
```

## 九、风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| MD 解析器复杂度高 | 使用成熟的 remark 库，定义清晰的模板格式 |
| 用户编辑破坏结构 | UI 层校验结构完整性，提供格式提示 |
| 分类准确率不够 | 调优 prompt，收集 bad case 迭代 |
| 成本增加 70% | 分类可用更便宜的模型，只对 candidate 消息调用 |
| 编辑冲突 | 乐观锁 + 队列，用户优先 |
