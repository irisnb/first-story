## Context

本项目是一个以知识库为核心、以模块画布为交互模型的剧本创作辅助工具。区别于传统线性剧本工具或纯大模型生成方案，本工具尊重创作的真实过程——无序、跳跃、迭代。

**核心约束（来自 CLAUDE.md，不可覆盖）：**
创作永远没有绝对的正确。工具是辅助者，不是裁决者。理论是工具不是答案，案例是参考不是模板。

## Goals / Non-Goals

**Goals:**
- 构建基于 RAG-Anything 的三合一知识库（理论 + 案例 + 视觉），本地 14B Q4 量化模型运行
- 实现模块画布交互模型，五大模块自由跳转，无固定流程
- 实现 Fountain 格式剧本编辑器，支持辅助模式和生成模式互通
- 实现渐进深度机制 + 过度构建防护 + 矛盾检测
- 实现分强度自检系统，大白话输出

**Non-Goals:**
- 第一版不做多人协作
- 不做云端同步（本地优先）
- 不做 AI 视频生成或分镜图生成
- 不做权限管理系统

## Decisions

### 1. 交互模型：模块画布 + 自由跳转（替代线性管线）

**选择**：五个核心模块（世界/角色/情节/主题/结构），用户可从任意模块开始，自由跳转，无固定顺序。

**理由**：
- 真实创作过程是非线性的——创作者可能从一个角色、一个场景、一个画面开始
- 6-P 等线性方法论有其价值，但作为"建议视角"而非"必须步骤"更合适
- 每个模块内部用五阶段循环（构思→实现→测试→反馈→迭代）推进

**替代方案**：线性管线（如 ScripThis 的 6-P 流程）——适合有经验的编剧从头开始，但限制了从任意入口进入的灵活性。

### 2. 知识库引擎：RAG-Anything（LightRAG 核心 + 多模态扩展）

**选择**：RAG-Anything 框架，14B Q4 量化模型本地部署（Ollama），约 9GB 显存占用。

**理由**：
- 知识图谱 + 向量混合检索，理解实体间关系（剧本知识的核心价值）
- 多模态能力为视觉参考库保留扩展空间
- 同一团队（HKU）维护的 LightRAG 生态，持续更新
- 14B 模型在 5060 Ti 16GB 上留 7GB 余量，可同时处理视觉任务

**替代方案**：
- 纯 ChromaDB：搭建极简单但不理解实体关系，不适合剧本知识的关联性
- 纯 LightRAG：功能完整但缺少多模态能力，升级需重做

### 3. 编辑器格式：Fountain

**选择**：Fountain 纯文本格式作为编辑格式，交付时渲染为 PDF/Word。

**理由**：
- 开放标准，纯文本可读可写
- 成熟的开源解析器生态（JS/TS/Python）
- 导出 PDF（afterwriting-labs）和 Word（pandoc/自建转换）技术成熟
- 避免从零构建复杂富文本编辑器

### 4. 渐进深度机制

**选择**：每个模块内容分三层——占位层（够驱动当前剧情）、可用层（足够支撑多场戏）、完整层（可用于外传/衍生）。

**理由**：
- 不强制"全部填完才能继续"，尊重创作的渐进性
- 工具在"浅层内容被当前创作触及时"温和提醒，而非在"未完成时"就催促
- 敏感度判断：无关不提 / 相关不急于记下来 / 关键在阶段结束时提醒

### 5. 过度构建防护 + 矛盾检测

**选择**：工具检测用户在单一模块的投入比例和内容之间的逻辑一致性。

**防护逻辑**：
- 当某模块内容量远超其他模块、且故事正文进展缓慢时，温和提醒
- 当检测到内容自相矛盾时，以询问语气提出（"你之前说…现在…是有意为之吗？"）
- 语气铁律：永远不阻止、不指责、不裁决

### 6. AI 辅助：Claude API（外部）+ 14B 本地模型

**选择**：知识库检索用本地 14B 模型（低延迟、零成本），复杂创作辅助可选择性调用 Claude API。

**理由**：
- 知识库检索是高频操作，本地模型保证响应速度和零成本
- Claude API 用于需要更强推理能力的场景（如深度自检、复杂情节分析）
- AI 功能需用户主动触发，避免不必要的成本

### 7. 存档系统：自动 + 手动 + 智能提醒 + 分叉

**选择**：四层存档体系——底层自动存档持续运行；中层手动存档点由用户创建；智能提醒在关键节点主动建议存档；高级分叉支持从任意存档点分支。

**存档结构**：
- 自动存档：持续后台运行，防崩溃/断电，用户零感知
- 手动存档：用户随时创建，命名 + 备注，类似游戏存档点
- 智能提醒：工具感知到大规模修改前主动提醒："你马上要大幅度调整第三幕了，要不要先存一下现在的版本？"
- 分叉：从任意存档点分叉出多条创作路线，同时尝试不同方向

**读档流程**：
- 读档时检测当前是否有未保存修改
- 若有 → 弹窗询问"要存一下再切换吗？"（存档后切换 / 不存档直接切换 / 取消）
- 若无 → 直接切换

**理由**：
- 自动存档消除数据丢失风险，零负担
- 手动存档 + 分叉给创作者"大胆尝试"的心理安全感
- 智能提醒帮用户判断"什么时候该存"，而非完全依赖用户自己的意识

### 8. 应用形态：Electron 桌面应用

**选择**：Electron + React + TypeScript，本地优先。

**理由**：
- 剧本内容敏感，本地存储优先
- 跨平台，创作者群体覆盖 Win/Mac
- 本地模型推理需要原生能力

## Risks / Trade-offs

- **14B 模型知识提取质量**：Q4 量化可能影响实体提取精度 → 关键文本可用 Claude API 做高质量提取，本地模型做日常检索
- **模块画布的导航复杂**：自由跳转可能导致用户迷失 → 提供"当前位置"指示和"建议下一步"可选引导
- **矛盾检测的误报**：可能把创作者的有意设伏判为矛盾 → 以询问语气呈现，由用户判断
- **Electron 体积**：打包后较大 → 对桌面应用可接受
