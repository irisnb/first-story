## Goals / Non-Goals

**Goals:**
- 用户能用 Fountain 写完一个几千字短剧本，全程顺手：写、存、不丢、关了再开还在、能导出
- 系统认得出角色（走 Fountain 结构，不猜）和事件（LLM 语义提取）
- 发现硬矛盾时递证据卡，用户能忽略/接受，系统记住偏好
- 修掉上次失败根因：提取从 rule-based 改为 Fountain 结构解析 + LLM 语义提取
- 固化防漂锚，保证 MVP 不偏离 AGENTS.md 愿景、不停在玩具版

**Non-Goals:**
- 完整六大 UI 模块（先一个编辑器 + 侧边证据卡）
- 自动语义合并版本树（只做历史 + 手动回退）
- 协作/多人、并发写入
- LightRAG 灵感检索、LangGraph 多 Agent、显式薄 Hub 调度层
- 心流自动识别、涟漪、伏笔、渐进提问、生成 Agent（代笔）
- 完整五层叠甲系统（只做一层温和措辞）
- Final Draft/Word 格式（只做 Fountain + 纯文本）

## Decisions

### D1: 角色识别走 Fountain 结构解析，不靠 LLM 猜
Fountain 剧本里角色名是结构化的——对白前一行居中大写即角色名。系统直接按 Fountain 语法确定角色集合，不做 NER 猜测。
- **为什么**：这正是上次失败的根因（`[一-龥]{2,4}` 把"房间""晚上"当人名）。Fountain 格式天然消除这个问题。
- **备选**：用 LLM 做 NER 识别角色——被否，多花钱、还引入幻觉，而结构解析免费且确定。

#### D1 决策快照（2026-06-20，实测后定，事实）
- **实测发现**：现成 pip 库 `jouvence`（唯一成熟的 Python Fountain 解析器）**无法识别中文角色名**——Fountain 标准靠"全大写行"判定角色提示行，中文无大小写，jouvence 把整段中文对白当成一个 action 块。只有用 Fountain 的 `@` 强制提示符（`@小明`）才正确切出"角色→对白"。另外 jouvence 还有两个硬伤：① `parse()` 有 bug（只有 `parseString()` 能用）；② 丢弃源字符偏移，而本 change 的 `source span 可定位` 要求每条 Fact 能定位回原文精确区间。
- **决定（任务 3.1 走"无合适再补核心子集"分支）**：自写最小 Fountain 结构解析器，核心子集 = 场景标题 / 角色提示行（含 `@` 强制） / 对白 / 动作 / 括号提示，**原生记录字符 offset**。不依赖 jouvence。
- **放弃了什么（请闻味道）**：中文角色名需要在角色行前加 `@`（如 `@小明`）才被识别为提示行。代价是用户多打一个字符。**替代方案——启发式猜测哪几行短中文是人名——正是上次失败根因，坚决不做。** 编辑器（Group 7）后续可加"一键插入 @"减少手动负担。
- **为什么接受这个代价**：宁可要一个"确定不猜、但需一个标记符"的可靠系统，不要一个"零负担但乱猜人名"的不可靠系统。这与灵魂清单 #1（不审判/不乱下定论）一致。

### D2: 提取 = 死规矩先行 + LLM 语义补全（两段式）
先用 Fountain 解析器抽出确定性结构（场景、角色、对白归属），把这些结构喂给 LLM 作上下文，LLM 只负责更难的语义层：事件摘要、角色状态变化、事实断言。
- **为什么**：落实文档第 291 行"死规矩先行 + 结果喂给 LLM"——确定的不花钱、不幻觉，只把 LLM 用在真正需要语义理解的地方，省钱 + 降误报。
- **备选**：整段正文直接丢给 LLM 全量提取——被否，贵、不稳定、难复现。

### D3: LLM 接口层抽象 + provider 可插拔
新增 `llm_provider` 服务，统一封装多模型调用（DeepSeek / OpenAI / Claude / 国产），从环境变量读 API key 与模型选择，内建 token 用量记录。提取服务依赖这个抽象，不直接绑某家 SDK。**MVP 默认 provider 为 DeepSeek**（用户提供 key 后接入），其余 provider 为可切换实现。
- **为什么**：用户指定默认 DeepSeek（性价比 + 中文能力）；抽象层让换 provider 不动提取逻辑，也对齐文档"成本工程化第一天纳入"。
- **备选**：直接调某家 SDK——被否，锁死 provider，换模型要改提取代码。

### D4: 提取/矛盾监控攒批触发，不每轮跑
对话 + 正文落盘每轮即时（不花钱）；LLM 提取 + 矛盾监控在停笔/保存后批量触发。
- **为什么**：对齐文档成本策略"不是每轮全员出动"；也避免边写边打断。
- **备选**：每次输入都提取——被否，token 成本不可控 + 打断创作。

### D5: 正文真相源仍是 event log，document revision 是其上的事件
保存正文 = 追加一个 `document.revised` 事件（含正文、source hash、source span）；历史/回退 = 重放到指定 revision。不新建独立存储。
- **为什么**：复用已验证的 event log 地基，天然支持历史/回退/不丢失（对齐"生长不覆盖"哲学）。
- **备选**：正文单独存文件——被否，破坏单一真相源，历史/恢复要另写一套。

### D6: 前端用 React + Vite
- **为什么**：回退的 MVP 就是这套，有现成参考可捡，省一半搭建时间。
- **放弃了什么**：换栈的"干净重来感"。这是可推翻的推荐，非事实——用户若有偏好可改。

### D7（防漂锚 · 灵魂清单）：每个 MVP 功能必须对回 AGENTS.md 不可动摇内核
以下五条是判定"有没有漂"的标尺。任何实现若违反其一，当场喊停、不继续：
1. **剧作老师，不审判**：系统不评判用户创意好坏，不下"你应该""必须三幕式"这类裁判腔。
2. **留余地，不下定论**：系统主动开口时只递证据、不给唯一答案；给"可能是故意的/先锋写法"退路。
3. **递证据，判断权归用户**：矛盾只列事实证据（evidence），不替用户决定怎么改。
4. **记住偏好，降权而非删除**：用户忽略某类提醒 → 写入 project_preferences 降权，不硬删、不永久消失。
5. **生长不覆盖**：所有改动只追加 event log，旧版本永不擦除，可回退。

落实映射：证据卡措辞过审 #1#2、ContinuityEvent 只存 evidence 不存结论 #3、忽略写降权规则 #4、document.revised 只追加 #5。

### D8（防漂锚 · 长大路标）：延后能力排队，不砍掉、不遗忘
以下能力**不是被删，是排队**，MVP 稳定后各拆独立 OpenSpec change 长回完整形态。列在此处防止沉到水底：
- 显式薄 Hub / 导播室调度层（单用户单进程暂用不上）
- Timing Policy 打扰风险判断 + 完整五层叠甲
- 心流模式自动识别（MVP 先手动开关）
- 顽固模式（重复 3 次升权）
- 涟漪检查、伏笔追踪、渐进式提问
- 生成 Agent（一键写/扩写/备选方向）
- LightRAG 灵感检索 + 知识库
- 记忆管理 Agent 完整版（跨项目用户记忆）
- Format Gate 完整导出门禁
- 版本分叉/对比/部分合并、完整六大 UI

终点仍是 AGENTS.md 那个有导播室的剧作老师；MVP 是它能站起来走路的第一版，后续能力往这个骨架上长进去，不推翻重来。

### D9: 矛盾检测的指代消解走独立 alias 归一层（身份与生死解耦）

灵魂功能（标杆案例：去世的姐姐又打电话）的矛盾，本质是**指代消解**问题——只有当系统认定"已故的姐姐 = 打电话的姐"是同一人时，alive+dead 冲突才成立。

- **为什么不让提取 LLM 一并判**：让同一次 LLM 调用既判身份又判生死，它会优先把故事讲圆、主动消解矛盾——实测把人拆成"已故姐姐"(dead) + "在世姐姐"(alive) 两个角色，冲突就自己消失了。
- **决定（B'）**：把"身份/别名归一"从状态提取**解耦**成独立 LLM pass（`alias_resolver`），它**只输出谁和谁是同一人**（canonical_name），**绝不碰生死**；状态仍由提取 LLM 按 fact 判（per-character），矛盾仍留在确定性层做 status 比对。检测器分组前查 alias 表把名字归一到 canonical——**查表，不靠关键词从结构猜语义**（守矛盾检测铁律）。
- **备选**：纯 prompt 工程在提取阶段加指代归一指令——被否，实测 0/9，太脆。

#### D9 决策快照（2026-06-20，真实 LLM 实测后定，事实）
- **实测命中率**：B' 把灵魂功能从纯 prompt 的 **0/9 抬到 8/9**（真实 deepseek-v4-pro，3 轮 ×3 次）。live API 全链路独立验证：保存矛盾文本 → 证据卡浮出（`status=queued`、`title=null`，只递证据无裁决）→ 忽略(queued→ignored) / 接受(queued→resolved) 均通。
- **已知缺口（请闻味道，不假装满分）**：剩余 ~1/9 漏报是 **LLM 非确定性**，无法靠 prompt 摁死，且不该靠堆 prompt 去摁（那正是脆弱性陷阱）。两类残留：
  1. **状态化命名 + alias 保守拒绑**：LLM 把人拆成"林婉的已故姐姐"/"林婉的在世姐姐"这种**名字本身就写死生死**的变体时，alias 层因"宁漏不误并、绝不碰生死"的保守设计而拒绝归并 → 漏。这是 B' 架构为"避免误并制造假矛盾"付出的**内在代价**，Oracle 评审已确认这条张力真实。
  2. **状态信号缺失**：打电话那句 LLM 偶发标 `unknown` 而非 `alive` → 不构成 alive+dead → 漏。
- **为什么接受这个代价**：误**并**（把两个不同的人并成一个）会凭空制造或抹掉矛盾，比漏**报**严重得多。保守优先符合 AGENTS.md"留余地不下定论、宁可漏也不乱下结论"。这条缺口按用户 2026-06-20 拍板**明确记为后续 spec 缺口**，不在 V1 强行补满。
- **后续长大路标**：模糊/状态化指代的稳定归并，留待 MVP 稳定后拆独立 change（候选方向：alias 表跨修订累积复用、亲属称谓唯一性的确定性预归并、人工确认指代的 UI）。

## Risks / Trade-offs

- **LLM 提取误报伤信任** → 死规矩先行降低误报面；MVP 收尾前用 20-30 个固定剧本片段评估集测误报率（信任验证节点 5）；用户可忽略且系统降权。
- **LLM 成本不可控** → 攒批触发（D4）+ 接口层 token 追踪（D3）+ 确定性结构不走 LLM（D2）。MVP 阶段先记录用量，预算熔断留到 Later。
- **LLM 调用失败/超时阻断创作** → 提取/矛盾是后台能力，失败时不得阻断写作；正文落盘与提取解耦，提取失败下次打开补提取（对齐文档"后台能力失败不阻断用户创作"）。
- **Fountain 解析覆盖不全（用户写非标准格式）** → MVP 支持核心 Fountain 元素（场景标题/角色/对白/动作）；非标准元素降级为普通动作行，不报错。
- **做着做着偏离原始愿景** → 灵魂清单（D7）+ 长大路标（D8）+ AGENTS.md 三条护栏（事实/推荐分标、推荐带"放弃了什么"、不可逆决策前喊停）。用户随时可喊"等一下"。
- **API key 泄露** → key 只从环境变量读，不入库、不回显、不写日志；外部调用走代理。
