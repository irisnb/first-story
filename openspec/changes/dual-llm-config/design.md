## Context

当前系统使用单一 LLM 配置（通过环境变量 `FIRST_STORY_LLM_API_KEY`），所有功能共用同一个模型。

问题：
1. 所有功能用同一个模型，成本无法优化
2. 用户无法在 UI 中配置 API
3. 无法切换不同模型

需求：
- 两个独立的配置槽位：`chat`（主聊天）、`utility`（分类/摘要等后台任务）
- 用户可以在设置中配置各自的 API 端点、模型、Key
- 支持多个配置方案切换

## Goals / Non-Goals

**Goals:**
- 支持两个独立的 LLM 配置：`chat` 和 `utility`
- API Key 安全存储（加密）
- 前端可配置（后续统一实现）
- 配置变更实时生效

**Non-Goals:**
- 前端 UI（本次只做后端）
- 支持超过两个配置槽位（V1 简化）
- 云端同步配置

## Decisions

### 决策 1：配置存储位置

**选择**：每个项目独立存储配置文件 `llm_config.json`

**理由**：
- 不同项目可能用不同的 API Key
- 简单直观，易于备份和迁移

**替代方案**：全局配置文件
- 否决理由：不支持项目级别的 API Key 隔离

### 决策 2：API Key 加密

**选择**：简单 base64 编码 + 项目级密钥混淆

**方案**：
```python
# 加密：base64(api_key + ":" + project_id)
# 解密：分离出 api_key
```

**理由**：
- V1 简单实现，防止明文直接可见
- 足够阻止普通用户误分享
- 不需要额外的加密库依赖
- 密钥管理简单（用 project_id 作为混淆因子）

**注意**：这不是真正的加密，只是混淆。V2 可升级为 Fernet。

### 决策 3：配置槽位

**选择**：两个固定槽位：`chat` 和 `utility`

```python
LLMConfigSlot = Literal["chat", "utility"]
```

**理由**：
- V1 简化设计
- 覆盖主要使用场景
- 后续可扩展

### 决策 4：配置继承

**选择**：项目配置 → 环境变量 → 报错

**流程**：
1. 检查项目配置文件
2. 如果没有，回退到环境变量
3. 如果都没有，返回错误提示配置

### 决策 5：配置 API 响应格式

**选择**：API Key 只返回后 4 位

```json
{
  "slot": "chat",
  "provider": "openai",
  "model": "gpt-4",
  "api_endpoint": "https://api.openai.com/v1",
  "api_key_preview": "****abcd"
}
```

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| API Key 泄露 | 加密存储 + 只返回后 4 位 |
| 加密密钥管理 | 从环境变量读取，文档说明 |
| 配置迁移 | 提供迁移脚本，从环境变量迁移到项目配置 |

## Migration Plan

1. 新项目：创建时自动生成空配置文件
2. 旧项目：首次访问时自动创建配置文件，从环境变量回退
3. 提供脚本：批量迁移环境变量到项目配置
