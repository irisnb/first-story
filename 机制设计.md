> - 整体结构见 `架构总览.md`
> - 哲学铁律见 `CLAUDE.md`
> - 架构共识总纲见 `AGENTS.md`

> **贯穿所有机制的最高原则**:系统永远"留余地",从不"下定论";判断权永远在用户手里。系统主动开口时闭嘴、压薄、只递证据;用户主动开口时全力配合。

## 1. 结构化事件契约

**要解决的问题**:后台 Agent 不能输出散乱自然语言给 Hub,否则无法稳定排队、恢复、叠甲。

**最终设计**:

- 所有后台发现必须先转成**结构化事件**,分三层信封:
  - **治理层**:event_id、版本、幂等键、状态(生命周期)
  - **语义层**:type、severity(P1-P5)、confidence、evidence、affected_modules
  - **投递层**:suggested_delivery_mode
- **去重**靠 event_id(幂等键),不靠内容比对
- **状态生命周期**用枚举管理(如:新建/排队/已展示/已忽略/已过期)
- **先锋艺术降权**用"方案C":保留原始判断 + 独立记录用户权重 + 可逆;触发靠**用户表态**,不靠系统判断"像不像艺术"
- **降权范围**用"方案丙":先窄后宽,用户反复表态才扩大;**降权有地板,永不沉默到 0**

**为什么这么定**:把"后台 Agent 无权决定打扰"这件事**编码进数据结构本身**(权限边界)。三层信封让权限、语义、投递各管各的,谁也越不了界。

**关键区分**:`severity`(严重程度,P1-P5)和 `confidence`(把握/信心)是**两个独立字段**,不可混为一谈。

## 2. 剧本格式理解

**定性**:这是一个**工具/按需能力**,不是新的常驻 Agent。被矛盾监控、提取 Agent、Format Gate 调用。

**要解决的问题**:剧本里有大量格式标记(回忆、梦境、旁白等),系统要读懂它们,才不会把"故意的手法"误判成"矛盾"。

**最终设计**:

- 标记分两类:
  - **现实位移类**(可豁免矛盾检查):回忆、梦境、闪回等——它们本来就"不在当前现实",时间线冲突可豁免
  - **呈现方式类**(不豁免):旁白、画外音等——它们仍在现实层,该查还得查
- 三档可信度:强标记 / 中标记 / 弱标记
- 中文 + 好莱坞两套格式都认
- Fountain 格式**不认**"回忆"这类标记,需自写解析层
- 输出用"方案乙":**中立报告**——只说"有什么 + 缺什么",**不建议**怎么补全

**为什么这么定**:格式检查是纯技术活,绝不涉及创作评判。中立报告守住了"不当格式警察"的铁律。

## 3. 用户记忆

**定性**:区别于知识库——这是"关于用户"的数据,驱动 Agent 的自适应行为。

**最终设计**:

- 分两层:
  - **用户级**(跨项目):创作节奏、时段偏好、术语熟悉度、互动习惯
  - **项目级**(单项目):各维度权重、已免疫的提醒类型、薄弱环节标记
- **忽略规则表**:独立于原始判断存储(原始判断不改,只是被忽略规则盖住)
- **三类信号双向升降权**:用户表态、行为模式、显式设置——都能让某类提醒升权或降权
- **跨项目升级**做成开关,默认"询问"(不自动把单项目偏好升成全局)
- **老规则沉底不删**:过期规则降到最低,但永不删除
- **透明 + 可导出**:
  - 便携人设包(自然语言)→ 可跨不同 LLM 使用
  - 完整存档(结构化)→ 用于完整恢复

**为什么这么定**:呼应全局原则——系统永不真正遗忘/丢弃,只调整显眼程度("降权而非删除")。

## 4. 叠甲厚度

**本质**:厚度是"压缩五层",**不是砍层数**。五层骨架(免责/事实/可能原因/后果/退路)永远都在,只是重甲全展开、中甲两三句、轻甲压成一句。

**三个影响因素**:用户档位 / 主观程度 / 严重性。

**最终设计(优先级 · 修正版)**:

- **系统主动发现问题时**:主观程度是**总闸**——越主观越压薄(防止系统主动说教)
- **用户主动要求时**:**用户优先**——要多少给多少(用户授权了就别藏)
- 区别在于**"谁发起"**:
  - 系统主动开口讲主观看法 → 危险,压薄
  - 用户明确说"我想听你的看法" → 这是授权,放开

**为什么这么定**:守住 CLAUDE.md 铁律(不拿主观看法当真理压人),同时满足"用户主动求助时别藏着掖着"(否则违反"绝对安全=绝对不好用")。

## 5. 版本分叉

**核心理念**:版本是"生长"不是"覆盖"——一棵树,节点永不丢弃。

**最终设计**:

- 支持:从任意历史节点分叉、随时回去、再分杈
- **比较**:V1 只做**简单比较**(文字并排对比);结构级深比较作后续增强
- **部分合并**:V1 **不做自动合并**。风险极高(蝴蝶效应/牵一发动全身)。
  - 若将来要做 → 高强度互动 + 用户逐步确认 + 在用户指引下完成
- **分支树整理**:方案乙——命名/标记主线,但**不限制**用户随便分叉

**为什么这么定**:呼应全天主线——系统永不真正删除任何东西。创作里没有"错的版本",只有"另一条没走的路"。自动合并必然产生逻辑断裂,所以交还用户。

## 6. 涟漪检查

**本质**:往前看的"预警"——改一处,提醒将被波及的连锁反应。

**与矛盾监控的区别**(防边界模糊):

- **矛盾监控** = 往回看,发现已经发生的冲突(事后纠错)
- **涟漪检查** = 往前看,提醒即将被波及的连锁(事前预警)

**最终设计**:

- **扩几圈**:方案丙——按改动核心程度动态决定。边角改动推 1 圈,伤筋动骨(主角生死/核心主题)多推几圈
- **防误报**:两道防线
  1. 只列事实关联,不下"必须改"结论(判断权还给用户)
  2. 复用用户记忆的"忽略规则",误报降权
- **触发**:核心设定变更 → 自动护航;小改动 → 用户可手动喊

**定性**:不是新怪物,是【矛盾监控的预测版 + 忽略规则 + 叠甲表达】三个现成零件的拼装。

## 7. 提取 Agent

**定位**:系统的"眼睛"。把用户人话 → 五大模块结构化数据,是所有后台能力的地基。没有它,后面全瞎。

**最终设计**:

- **两挡变速**:
  - **快挡**(实时瞟一眼):低成本、不卡顿、只默默记
  - **慢挡**(空闲深度精读):大模型精读,补全补准
  - ★ **只有慢挡精读后才允许"提醒用户"**(防快挡粗数据误报)
- **提取粒度分层**:
  - 实体 + 关系 → 主力做深(系统擅长)
  - 因果 → 只做"灵感辅助" + "观众镜子",**绝不进纠错主链路**(系统不擅长)
  - ★ **"观众镜子"特性**:连全程深度分析的 AI 都没看懂因果 → 普通观众更不可能懂 → 是个值得用户知道的强信号;但只作温柔的"观众困惑反馈",绝不评判"你写错了"
- **看错兜底**:三道——标把握度 / 用户可改可逆 / 记忆学习用户表达习惯
  - ★ **技术现实修正**:把握度(confidence)不是 LightRAG 原生白送的,需自己包一层
  - ★ **低把握度提取 → 先进"待复核队列",不直接写主状态**(慢挡精读/用户确认后才正式落地)
- **用户控制分两层**:
  - **体验挡位**(给用户,大白话):沉浸 / 平衡 / 助手 + 心流总开关
  - **技术挡位**(藏后台,自动适配):快慢挡、模型选择,用户不碰

**为什么这么定**:实体/关系系统擅长,可放心做深;因果提取在内向型叙事会退化(LightRAG 已知限制),所以只用来"启发",绝不用来"纠错"——否则必然狼来了。

**技术验证**:LightRAG 自带"快入口 + 慢处理"机制(`apipeline_enqueue` + `apipeline_process`),两挡变速业界有成熟实现;角色化 LLM 可分别配置(EXTRACT/QUERY/KEYWORDS/VLM),支持成本分层;但其原生 schema 无 confidence 字段,需自建 wrapper。

## 8. 心流识别

**定位**:优先级最高的静默状态。判定心流后,系统几乎全闭嘴。

**核心价值排序**:正在流淌的创作 > 修正错误。"bug 能补,灵感断了回不来"——连"善意的纠错"都不许打断创意。

**最终设计**:

- **怎么判断**:组合信号,且**分级**——
  - **强信号**(连续产出 + 不回改 / 用户手动开关)→ 说了算
  - **中信号**(历史时段习惯)→ 做加权
  - **弱信号**(不理系统)→ 只在强信号成立时加固,**绝不单独定案**
- **判断错了**:自动判断 + 手动一键开关 双保险
  - 心流只挡"系统主动说话",**绝不挡"用户主动求助"**(用户主动开口 → 立刻响应)
- **怎么退出**:系统观察到明显停顿 → 温柔自动退出 + 用户可手动
  - 退出时把攒下的提醒"**重新评估**"(过期的不再提),而非一股脑倒出

**为什么这么定**:任何单一信号都会骗人(打字快可能是烦躁、不理系统可能是去倒水),组合分级才靠谱。误判哪边都难受,所以兜底机制比判断本身更重要。

## 9. 渐进式提问

**本质**:ChatUI 的灵魂。不是收集信息,是"帮用户想"——像剧作老师顺着话头往深里带。

**最终设计**:

- **下一个问什么**:两层叠加
  - **客观缺口**(五大模块完成度,复用提取 Agent 数据)→ 决定往哪个方向补
  - **用户记忆**(喜好/习惯,复用记忆文档)→ 决定用什么方式问、问不问
  - 既补最缺的,又顺着这个人的脾气
- **问到哪儿停**:分两级
  - **总开关**(心流/沉浸挡)→ 最高优先,一关全静默
  - **细则刹车**(总开关没关时):用户开始自己写 / 这方向够用了 / 用户不耐烦
  - ★ **硬上限**:哪怕最积极的助手挡,一轮最多碰 **3 个方面**,绝不超
- **不想被问**:频率跟体验挡位走(沉浸/平衡/助手)+ 心流总开关兜底
  - 不自己造规则,听 Timing Policy

**为什么这么定**:渐进提问是"邀请"不是"考试",用户随时可不答、可自己写。连"用户主动要积极引导"都装了刹车(3 个方面硬上限),因为热情过头也是打扰。

## 10. 顽固模式

**核心智慧**:不靠"读心猜你是否故意",靠"看你坚持不坚持"。反复出现且不解释 = 用户用行动说"这是我的设定" → 系统转向尊重。

**最终设计**:

- **升级台阶**:
  1. **轻甲·随口一问**:极轻,像朋友随口一提,可完全无视
  2. **中甲·认真一点**:稍正式,仍给"有意为之"留足退路
  3. **重甲·最终警告(郑重告别)**:语气稍重,说清后果,明确"这是最后一次主动提它"
  4. 之后不改不解释 → **转向尊重**,记入项目记忆,不再主动提
  - ★ 最终警告重在"郑重",不在"施压";是句号,不是最后通牒
- **红灯**:保留,但只给"几乎不可能是创意"的硬伤(如拼错名字)
  - 用"主观程度"判断红不红灯;心流时红灯也只亮卡片,**绝不插队**
- **转向后不焊死**:尊重为主,用户自己回头困惑时,系统才极轻接一次

**为什么这么定**:系统没法读心,分不清"写错了"和"故意的"。用"时间和重复"代替"读心"——你坚持,就是你要的。无限升级施压会变成打压创意的格式警察,踩铁律红线,所以升级到"郑重告别"为止,然后尊重。

## 跨机制的零件复用图

今天很多机制不是从零造的,而是复用已定零件拼装的。这张表帮你看清依赖:

| 机制 | 复用了哪些已定零件 |
|------|--------------------|
| 涟漪检查 | 矛盾监控(预测版)+ 忽略规则 + 叠甲表达 |
| 渐进提问 | 提取 Agent(缺口数据)+ 用户记忆 + Timing Policy + 心流 |
| 提取 Agent | 把握度(叠甲)+ 用户可改可逆(记忆)+ 记忆学习 |
| 顽固模式 | 主观程度(叠甲那把尺子)+ 项目记忆 + 心流 |
| 格式理解 | 被矛盾监控 / 提取 / Format Gate 调用 |

**这说明系统是一个有机整体,而非一堆孤立功能。** 一个零件(如"主观程度""忽略规则")往往在多个机制里反复发挥作用。
